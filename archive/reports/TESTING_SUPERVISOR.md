## Детальное тестирование

### Тест 1: Structured Error (легковесная обработка)

**Цель:** Проверить что structured errors обрабатываются БЕЗ LLM

```python

# Например, при неверном параметре:

result = await executor.execute({
    "action": "navigate",
    "params": {}  # ← нет URL!
})

# Ожидаемое поведение:
# - SupervisorAgent логирует ошибку (0 токенов)
# - Возвращает result с suggestion
# - НЕ вызывает LLM
```

**Как проверить:**
1. Запустить агента: `python main.py`
2. Дать задачу без нужных параметров
3. Проверить что:

   - Пользователь видит понятное сообщение

### Тест 4: Экономия токенов

**Цель:** Убедиться что Supervisor НЕ тратит токены на успешных операциях

**Метод:** Логирование

1. Добавить временный лог в `supervisor_agent.py`:
```python
async def supervised_execute(self, executor, action):
    print("[SUPERVISOR] Observing action...")  # ← Добавить
    try:
        result = await executor.execute(action)

        if result.get("status") == "error":
            print("[SUPERVISOR] Structured error - no LLM")  # ← Добавить
            return await self._handle_structured_error(result, action)

        print("[SUPERVISOR] Success - sleeping 😴")  # ← Добавить
        return result

    except Exception as e:
        print("[SUPERVISOR] Runtime error - calling LLM! 🚨")  # ← Добавить
        return await self._handle_runtime_error(e, action)
```

2. Запустить агента и дать простую задачу:
```
python main.py
User: "Открой google.com"
```

3. Наблюдать:
```
[SUPERVISOR] Observing action...
🔧 Выполняю: navigate
   ✓ Результат: success
[SUPERVISOR] Success - sleeping 😴
```

**Ожидаемое:** Ни одного вызова LLM для успешных операций

### Тест 5: Интеграция (E2E)

**Полный сценарий:** Production → Error → Sandbox → Fix

1. **Production:** Работаем с агентом
```bash

User: "Найди на странице google.com кнопку 'НесуществующаяКнопка'"
# → Ошибка зафиксирована
```

2. **Анализ:** Смотрим что попало в логи
```bash
cat data/errors/session_*.json | jq '.errors[0]'
```

3. **Sandbox:** Воспроизводим
```bash
python main.py --sandbox
# → Sandbox пытается воспроизвести и найти альтернативы
```

4. **Отчёт:** Смотрим рекомендации
```bash
cat data/sandbox_reports/sandbox_report_*.json | jq '.recommendations'
```

5. **Fix (manual):** Если sandbox нашёл альтернативу - применяем её в коде

## Проблемы и решения

### Проблема: Supervisor не создаёт файлы

**Причина:** Нет прав на запись в `data/errors/`

**Решение:**
```bash
mkdir -p data/errors
chmod 755 data/errors
```

### Проблема: Sandbox не находит ошибки

**Причина:** Ошибки сохранены в другом месте или формате

**Проверка:**
```bash
# Проверить что файлы существуют
ls -la data/errors/

# Проверить формат
cat data/errors/session_*.json
```

**Решение:** Убедиться что Supervisor работал в production mode

### Проблема: LLM не возвращает валидный JSON

**Проявление:** В логах `"LLM не вернул валидный JSON, использую fallback"`

**Причина:** LLM вернул текст вместо JSON

**Решение:** Уже обработано fallback'ом, но можно улучшить промпт в `_build_error_context()`

## Метрики успеха

✅ **Production mode работает если:**
- Structured errors логируются без LLM вызовов
- Runtime errors анализируются с LLM (~500 токенов)
- Session summary создаётся при завершении
- Пользователь видит понятные сообщения

✅ **Sandbox mode работает если:**
- Читает ошибки из `data/errors/`
- Генерирует 3+ сценария на каждую ошибку
- Пытается воспроизвести и найти альтернативы
- Создаёт детальный отчёт с рекомендациями

✅ **Экономия токенов достигнута если:**
- На 10 успешных действий: 0 токенов
- На 1 structured error: 0 токенов
- На 1 runtime error: ~500 токенов
- Итого в худшем случае: 500 токенов вместо краша
